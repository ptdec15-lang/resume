---

```
┌──────────────────────────────────────────────────────────┐
│                                                          │

│                                                          │
│   第一层  模块内分析        ← 逐模块，从数据到业务判断     │

│     ├─ ③ 机械归因                                       │

## 第零层：上下文锚定

> **原则：先搞清楚"已知原因"，再去找"未知异常"。**

### 0.1 本周特殊事件清单

列出所有**已知的**可能影响数据的事件，避免在后续分析中把"已知变更"当成"异常"去归因。

| 类型 | 检查项 | 示例 |
| :--- | :--- | :--- |
| 实验变更 | 本周是否有新实验上线/下线/调整比例？ | "大额PUSH增发实验exp2扩量至10%" |
| 运营活动 | 是否有促销、签到加码、文案更新？ | "周三上线嘉年华活动" |
| 技术变更 | 是否有版本发布、服务迁移、Bug修复？ | "周四修复了挂件加载超时问题" |
| 外部因素 | 节假日、竞品动作、季节性、市场大盘？ | "巴西嘉年华假期" |

### 0.2 上周Action Item回顾

| 上周提出的动作 | 是否执行 | 本周数据是否有反馈 |
| :--- | :--- | :--- |
| （填入上周建议） | 是/否/部分 | 对应指标变化 |

**为什么要做这一步**：如果上周建议了"优化push文案"，本周push点击率翻倍，那这个变化就是"已归因"的，不需要重复分析。

### 0.3 基准预期设定

在看数据之前，先基于上周数据和已知事件，对关键指标设一个"预期范围"：

> 示例：上周DAU贡献量608万，本周无重大变更，预期范围 600-615万（±1.5%）

超出预期的才需要深入归因，在预期内的属于正常波动。

## 第一层：模块内分析

> **原则：每个模块回答一个业务问题，分析时始终围绕这个问题。**

对看板中的每个模块，依次执行以下5步：

### 步骤①：趋势扫描 — "这个模块的数字长什么样？"

**做什么**：快速浏览本模块所有指标的本周值、上周值、周环比、趋势图。

**输出格式**：

| 指标 | 本周 | 上周 | 周环比 | 趋势形态 |
| :--- | :--- | :--- | :--- | :--- |
| 指标A | xxx | xxx | +x.x% | 平稳/上升/下降/波动 |
| 指标B | xxx | xxx | -x.xpp | 持续下滑（连续N周） |

**注意**：这一步只描述现象，不做判断。用数字说话，不用形容词。
- 说"下降0.18pp"，不说"略有下降"
- 说"连续3周下滑，累计降幅2.5pp"，不说"持续恶化"


---

### 步骤2：机械归因 — 

见归因方法论.md

**归因四板斧**：

```
                    指标异动
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
      拆公式         拆时段        拆渠道
   (分子/分母)    (哪天/哪时段)   (哪个来源)
          │            │            │
          ▼            ▼            ▼
       定位到具体变动项
          │
          ▼
      查事件表（步骤0.1）
   是否有已知原因可解释？


---

### 步骤④：关系分析 — "模块内的指标之间什么关系？"

**做什么**：分析本模块内多个指标之间的结构性关系。

**三种常见关系模式**：

#### 模式A：漏斗关系
指标之间存在转化链路，前一步的结果是后一步的输入。

```
曝光UV → 点击率 → 进入UV → 完成率 → 完成UV
```

**分析方法**：
- 计算各环节转化率
- 找到转化率最低/下降最大的环节 = 瓶颈
- 漏斗中任何一环的变化都会传导到后续环节


#### 模式B：总量-均值矛盾
总量指标和人均/设备均指标方向不一致。


**分析方法**：
- 总量 = 人均 × 人数
- 总量涨 + 人均降 → 纯靠人数驱动，质量在稀释
- 总量涨 + 人均也涨 → 量质齐升，最健康的状态

#### 模式C：对照组比较
实验组和对照组的差异反映策略的增量效果。


**分析方法**：
- 差值为正 → 策略有正向效果
- 差值趋势走低 → 策略效果衰减
- 差值在某些维度为负 → 策略存在副作用

**输出**：用一句话描述本模块内指标的关系结论。

---

### 步骤⑤：业务判断 

**做什么**：将步骤①-④的数据分析翻译为业务语言。


**判断模板**：

> 关键信号：[最重要的1-2个发现]  
> 业务含义：[对业务目标意味着什么]

## 第二层：跨模块综合分析

> **原则：模块之间的"矛盾"比"一致"更有价值。**

完成所有模块的独立分析后，将各模块结论放在一起审视。

### 步骤①：矛盾提炼 — "哪些结论互相打架？"

**做什么**：寻找不同模块之间"本应同向变动但实际背离"的指标对。

**寻找矛盾的方法**：

| 检查维度 | 本应关系 | 如果出现反向，意味着 |
| :--- | :--- | :--- |
| 北极星 vs 渗透 | DAU涨 → 渗透应涨或至少持平 | 增量用户未进入核心场景 |
| 北极星 vs 消费深度 | DAU涨 → 时长/播放应涨 | 增量用户是浅层活跃 |
| 渗透 vs 触达漏斗 | Push效果好 → 渗透应涨 | 触达后的承接链路有问题 |
| 消费深度 vs 成本 | 时长涨 → 收入/回收应涨 | 时长增量不产生广告价值 |
| 总量 vs 人均 | 总量涨 + 人均也涨 = 健康 | 总量涨 + 人均降 = 靠量撑，质量稀释 |

**输出格式**：

```
矛盾 #1：[模块A结论] ←→ [模块B结论]
         ↳ 暗示：xxxxxxxxx
```

**实战示例**：

```
矛盾 #1：DAU贡献量+1.14% ←→ 任务中心渗透率-0.18pp
         ↳ 暗示：新增DAU不是任务驱动型用户

矛盾 #2：APP总时长+12.9% ←→ 设备均播放时长-15.1%
         ↳ 暗示：激励拉来的是"打卡即走"的浅层用户

矛盾 #3：Push点击率翻倍 ←→ 365天回收率持续下滑
         ↳ 暗示：触达效率提升了，但拉来的用户不产生收入
```

---

### 步骤②：因果链重建 — "把矛盾串成一个故事"

**做什么**：将多个矛盾串联，还原出一条完整的因果链条。

**方法**：从最底层的原因出发，逐步向上推导到最终结果。

```
根因
  └→ 中间环节1
       └→ 中间环节2
            └→ 最终症状
```

**实战示例**：

```
激励策略吸引了大量浅层用户（根因）
  └→ 这些用户进APP只做任务，不消费内容（行为表现）
       ├→ 设备均播放时长为负（消费深度信号）
       ├→ 广告曝光少 → 回收率下滑（收入信号）
       └→ DAU总量涨但渗透率降（渗透信号）
            └→ 成本不变但回收变差 → 成本效率恶化（成本信号）
```

**检验方法**：一条合格的因果链应满足：
1. **逻辑自洽**：每一步推导都说得通
2. **数据支撑**：每一步都有对应的指标为证
3. **可证伪**：能想到"如果这个判断错了，哪个指标应该不一样"

---

### 步骤③：核心叙事 — "用1-3句话说清楚本周怎么了"

**做什么**：将因果链浓缩为一段可以在周会开场30秒内讲清楚的总结。

**格式**：

> **[结果描述]，但[矛盾信号]，[根因判断]，[风险/机会判断]。**

**示例**：

> 本周DAU"量率齐升"，但渗透率微降+回收率持续下滑+设备均时长为负，信号指向增量用户偏浅层——"来了但没真正用"，需警惕DAU增长的可持续性。

**自检**：好的核心叙事应该让听众立刻知道——
- 现在好不好？（DAU量率齐升 → 表面上好）
- 有什么隐患？（三个矛盾信号 → 底层不好）
- 为什么？（增量用户偏浅层 → 原因）
- 接下来怎么办？（需警惕可持续性 → 引出讨论）

## 第三层：决策支撑


### 步骤①：风险评估

**做什么**：对核心矛盾/趋势做外推，评估"不作为"的后果。

| 趋势 | 当前值 | 变化速度 | 红线 | 触达红线时间 | 后果 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 365天回收率下滑 | 12.7% | ~1pp/周 | 10% | 约3周 | 逼近成本倒挂 |
| 渗透率下降 | 18.82% | ~0.2pp/周 | 17% | 约9周 | 任务中心规模萎缩 |


---

### 步骤③：待验证假设 — "我们还不确定什么？"

**做什么**：诚实列出本周分析中的推测性判断，明确下周需要哪些数据来验证。

| 假设 | 验证方法 | 下周关注指标 |
| :--- | :--- | :--- |
| "push点击率翻倍源自大额PUSH实验" | 拆分实验组vs对照组的push点击率 | 大额PUSH组单独点击率 |
| "增量用户是浅层用户" | 看新增用户的次留和7留 | 新用户留存率 |

**为什么需要这一步**：
- 避免过度自信——数据分析常有多种解释，明确标记"推测"和"确认"
- 为下周分析埋线索——下周第零层的"Action Item回顾"直接对接这里
- 形成分析的**闭环**：假设→验证→新假设→再验证

## 附录：常见分析陷阱

| 陷阱 | 表现 | 纠正 |
| :--- | :--- | :--- |
| **数据鸵鸟** | 只报好消息，回避坏数据 | 好坏必须同时呈现，预警信号不能省 |
| **均值陷阱** | 只看周均值，忽略日内波动 | 均值异常时必须下钻到日级/小时级 |
| **因果倒置** | "渗透率降了所以DAU不好" | 渗透率是DAU的驱动因素，不是结果。方向搞反会导致行动错误 |
| **忽视分母** | "曝光UV降了所以渗透率降了" | 渗透率=UV/DAU，可能UV没降但DAU涨了，分母效应 |
| **归因确认偏差** | 先有结论再找数据支持 | 先看数据，再形成判断。如果因果链中有一环数据不支持，整个链条要重新审视 |
| **过度归因** | 对每个±0.1pp的波动都写一段分析 | 用步骤②分级筛选，不是所有波动都值得归因 |
| **模糊建议** | "建议持续关注""建议优化" | 每条建议必须可执行：做什么、谁来做、什么时候看效果 |
