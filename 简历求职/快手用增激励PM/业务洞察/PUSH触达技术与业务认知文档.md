# PUSH 触达技术与业务认知文档


## 一、技术名词解释（非技术人员版）

### 1.1 数据存储与处理

| 术语 | 通俗解释 | 类比 |
| :--- | :--- | :--- |
| **Hive 表** | 一种存储海量数据的"超大号 Excel 表格"，存放在公司大数据平台上。适合存储和分析历史数据，但查询速度相对较慢（分钟级），不适合实时场景 | 图书馆的档案柜——存得多、查得全，但要花时间翻找 |
| **Hive 分区** | 把 Hive 表按某个维度（如日期、小时）切成多个"抽屉"，查询时只翻对应的抽屉，大幅提高效率 | 档案柜按年份/月份分隔的隔板 |
| **Kafka** | 一种高速"消息传送带"系统。数据从一端放入，另一端实时消费。擅长处理大量实时数据流，保证消息不丢失、按顺序到达 | 寿司店的传送带——厨师放上去，客人按顺序取走 |
| **Topic** | Kafka 中的"频道"概念。不同业务的数据放在不同 Topic 里，互不干扰 | 传送带上不同颜色的盘子，代表不同种类的寿司 |
| **QPS** | Queries Per Second，每秒处理的请求数量，衡量系统吞吐能力的指标 | 一条高速公路每秒能通过多少辆车 |

### 1.2 推送相关

| 术语 | 通俗解释 | 类比 |
| :--- | :--- | :--- |
| **PUSH 推送** | 服务端主动向用户手机发送通知消息，无需用户打开 APP | 快递员主动把包裹送到你家门口 |
| **直发 PUSH** | 由业务方直接指定"发给谁、发什么、什么时候发"，不经过系统混合排序，确保精准送达 | 指定快递——你亲自写好收件人地址，快递公司直接送 |
| **混排 PUSH** | 多个业务的 PUSH 进入统一排序池，系统根据用户偏好、优先级等综合排序后择优下发 | 拼车服务——系统把多个乘客的需求统一调度，选最优路线 |
| **频控** | 频率控制，限制向同一用户发送 PUSH 的次数，防止骚扰用户导致关闭通知权限 | 小区门卫限制同一个推销员每天只能进来一次 |
| **时效性 PUSH** | 一种带有时间紧迫感样式的 PUSH，视觉上更显眼，点击率通常更高 | 标注了"限时优惠"红色标签的传单 |
| **Deeplink** | 一种特殊链接，点击后能直接跳转到 APP 内的指定页面（而非仅打开 APP 首页） | 精确到楼层和房间号的地址，而不只是一个小区名 |

### 1.3 实验与策略

| 术语 | 通俗解释 | 类比 |
| :--- | :--- | :--- |
| **AB 实验** | 将用户随机分成若干组，每组看到不同的策略，通过数据对比找出最优方案 | 同一款药的临床试验——一组吃药、一组吃安慰剂，对比疗效 |
| **通配符** | PUSH 文案中的占位符（如 `${amount}`），系统下发时自动替换为每个用户的真实数据 | 合同模板中的"甲方___"空格，签约时填入实际名字 |
| **触发源** | PUSH 下发的数据来源配置，定义"从哪张表取数据、筛选什么条件、发到哪个 Kafka Topic" | 自动化流水线的第一道工序——原料从哪里来 |

### 1.4 数据链路

| 术语 | 通俗解释 | 类比 |
| :--- | :--- | :--- |
| **Hive2Kafka 作业** | 一种自动化数据搬运任务：定时把 Hive 表中的数据读出来，转发到 Kafka 消息队列中 | 仓库管理员每天定时把档案柜里的订单取出来，放到传送带上 |
| **ETL** | Extract-Transform-Load（抽取-转换-加载），从源头取数据、清洗加工、写入目标系统的标准流程 | 食材采购→洗菜切菜→上桌的厨房流程 |
| **RPC 服务** | Remote Procedure Call，远程过程调用。系统之间互相请求数据的方式 | 打电话给另一个部门要数据 |
| **限流** | 控制系统每秒处理的请求数量上限，防止系统过载崩溃。超出限流的请求会进入低优先级队列或被丢弃 | 商场限流——一次只允许进 500 人，多的在外面排队 |

## 二、PUSH 下发的主要方式与逻辑

### 2.1 两种核心下发模式

```
┌─────────────────────────────────────────────────────────────┐
│                    PUSH 下发方式对比                          │
├──────────────┬──────────────────┬───────────────────────────┤
│              │     直发 PUSH     │        混排 PUSH          │
├──────────────┼──────────────────┼───────────────────────────┤
│ 控制权       │ 业务方完全控制    │ 系统统一调度               │
│ 发送对象     │ 业务方精确指定    │ 系统综合筛选               │
│ 发送时间     │ 业务方指定时间窗口│ 系统选择最优时机            │
│ 文案内容     │ 业务方定制        │ 多业务竞争，系统择优        │
│ 频控         │ 需业务方自行管理  │ 系统全局频控               │
│ 到达确定性   │ 高（直接下发）    │ 中（可能被其他PUSH挤掉）    │
│ 适用场景     │ 时效性强、金额大  │ 日常运营、内容推荐          │
└──────────────┴──────────────────┴───────────────────────────┘
```

### 2.2 直发 PUSH 的完整链路

以签到大额 PUSH 增发需求为例，直发 PUSH 的端到端数据流如下：

```
[1] Hive 表（存储用户签到行为数据）
     │
     │  定时 Hive2Kafka 作业
     │  （每日北京时间 18:00 触发，读取当日签到数据）
     ▼
[2] Kafka Topic（实时消息队列）
     │
     │  PUSH 服务消费 Kafka 消息
     ▼
[3] PUSH 下发引擎
     │  ├─ 用户筛选（地区、签到状态、AB 分组）
     │  ├─ 频控校验（当日是否已推送）
     │  ├─ 文案组装（通配符替换为真实金额）
     │  └─ 时效性样式渲染
     ▼
[4] 手机系统推送通道（APNs / FCM）
     │
     ▼
[5] 用户手机收到 PUSH 通知
     │
     │  用户点击
     ▼
[6] Deeplink 跳转至任务中心签到页
```


## 三、Hive + Kafka 模式详解

### 3.1 为什么选择"Hive 表 + Kafka"这种架构？

这不是随意的技术选择，而是根据 PUSH 推送业务的特性做出的合理决策：

| 业务需求 | 技术方案如何满足 |
| :--- | :--- |
| 需要大批量处理用户数据（百万级） | Hive 擅长海量数据的批处理和筛选 |
| 需要在指定时间窗口推送 | Hive2Kafka 作业可配置定时调度 |
| 需要实时、高吞吐地逐条下发 PUSH | Kafka 天然支持高并发消息分发 |
| 需要控制推送速率，避免系统过载 | Kafka 消费端 + Hive2Kafka 限流配置可精确控速 |
| 需要数据可追溯、可回查 | Hive 表按日期分区，历史数据完整保留 |

### 3.2 这种模式的优缺点

**优点**：

1. **批量与实时的最佳结合**：Hive 负责"谁需要收到推送"的离线筛选，Kafka 负责"把推送实时送出去"的在线下发，各司其职
2. **可控性强**：通过 Hive2Kafka 的限流配置（QPS 限制）和调度时间配置，能精确控制推送速率和时间
3. **成本低**：利用已有的大数据基础设施（Hive 集群），不需要额外开发实时计算系统
4. **容错性好**：Kafka 消息持久化，即使下游 PUSH 服务短暂故障，消息不会丢失，恢复后继续消费
5. **可观测**：Hive 表数据可随时查询验证，Kafka 有消费进度监控，链路透明

**缺点**：

1. **实时性有限**：依赖 Hive 分区的更新频率（小时级），无法做到秒级响应用户行为
2. **链路较长**：数据从 Hive → Kafka → PUSH 引擎 → 手机，任何一环出问题都会影响推送
3. **时区处理复杂**：Hive 分区按北京时间，推送按巴西时间，需要在多个环节做时区转换

### 3.3 适用场景判断

```
你的推送需求适合用"Hive + Kafka"模式吗？

问题1：推送的目标用户需要通过复杂条件筛选？
  → 是 → 适合，Hive 擅长复杂条件批量筛选

问题2：推送有明确的时间窗口？
  → 是 → 适合，Hive2Kafka 可配置定时调度

问题3：推送量级大？
  → 是 → 适合，Hive + Kafka 组合能轻松处理百万级

问题4：推送需要秒级响应用户行为？
  → 是 → 不适合，应选择实时事件驱动方案
  → 否 → 适合
```


### 4.3 直发 vs 混排的选择建议

| 判断维度 | 选直发 | 选混排 |
| :--- | :--- | :--- |
| 推送时效性 | 强（必须在指定时间窗口到达） | 弱（当天送到即可） |
| 业务优先级 | 高（核心转化节点） | 中低（日常运营） |
| 文案个性化 | 需要精确的金额/天数等动态数据 | 通用文案即可 |
| 到达率要求 | 高（不能被其他 PUSH 挤掉） | 可接受一定损耗 |
| 频控管理 | 愿意自行管理 | 希望系统统一管理 |

## 五、Hive2Kafka 作业配置要点

### 5.1 核心配置项

在实际项目中，Hive2Kafka 作业需要关注以下配置：

| 配置项 | 说明 | 示例值 |
| :--- | :--- | :--- |
| Hive 库名 | 数据所在的 Hive 数据库 | `os_growth_ad` |
| Hive 表名 | 具体的数据表 | `oversea_push_shared_user_upload_photo_di_v2`（示例） |
| 目标 Kafka 集群 | 消息发送到哪个 Kafka 集群 | `sgpcc-common1` |
| Topic 名称 | 消息发送到哪个频道 | `oversea_push_notify_trigger_access` |
| 数据过滤条件 | Hive 读取时的 WHERE 条件 | `p_date = '当日日期'`、`country_code = 'br'` |
| QPS 限速 | 每秒发送到 Kafka 的最大消息数 | 根据业务场景配置，默认上限 3000 |

### 5.2 限流机制

系统为每个业务场景单独限流，需要注意：

- **默认 QPS 上限**：3000。超出后数据进入低优先级队列
- **低优先级队列**：以固定 QPS 尽力消费，超时消息会被直接丢弃
- **主动限流方式**：
  - 在 Hive2Kafka 作业的"通道配置 → 限速配置"中指定 QPS
  - 在"调度配置"中调整调度时间，实现错峰发送
- **延时告警**：可在 Hive2Kafka 作业中配置延时告警，报警级别和时间由业务方自定义
