# 什么是真正的产品能力？

## 一、根本区别：做事 vs 建系统

产品经理的本质工作是**建系统，而不是做事情**。

| 角色 | 类比 | 工作方式 |
|------|------|----------|
| **运营** | 消防员 | 哪里着火扑哪里，日常巡检，发现异常就报警 |
| **产品** | 建筑师 | 设计一套消防系统，让火灾自动被检测、自动被扑灭，新楼也能复用 |
| **策略产品** | 城市规划师 | 不是设计一栋楼的消防，而是制定整个城市的消防规范和标准 |

**核心检验标准**：

> 产品能力 = 你能不能建一个系统，让这个系统在你走了之后还能跑，换个场景还能用。

---

## 二、四大核心产品能力

### 能力一：产品设计复用能力

**定义**：你做过的东西，能不能换个场景再用一遍？


**正面**：设计的是一套可复用的框架，输入端可替换，框架本身通用。
- 像搭积木一样快速组合，服务多个目标场景
- 有完整的产品能力体系

输入内容 ---- 产品框架 ---- 输出结果
---

### 能力二：竞品分析与迁移能力

**定义**：能不能把别人的好东西"偷"过来，还能因地制宜地改？

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1. 看到 | 竞品做了什么 | 大家都会的基本功 |
| 2. 理解差距 | 为什么人家能做我们不能 | 是资源差、用户不同、还是技术限制？ |
| 3. 识别可迁移能力 | 哪些底层逻辑可以搬过来 | 抽象出通用的产品模式 |
| 4. 结合自身落地 | 搬过来之后怎么改 | 适配自己的业务约束 |


---

### 能力三：用户理解与决策能力

**定义**：当用户角色很复杂的时候，能不能搞清楚到底该为谁做产品？

解决方法：用增量思维判断业务价值进而判断资源投入

**能力体现**：不需要选对，但要有逻辑——能说清楚"我选了哪条路、为什么"。

---

### 能力四：需求判断与拒绝能力

**定义**：100个人找你提需求，能不能说清楚哪些做、哪些不做、为什么？

**需求的三层过滤器**：

```
第一层：真问题还是伪问题？
  → 数据没异常，但业务方"感觉"不好 = 伪问题

第二层：值不值得产品化解决？
  → 只出现一次 → 写个脚本跑一下，不用做产品
  → 反复出现 → 值得建系统

第三层：现在做还是以后做？
  → 收益大、成本低、和当前目标一致 → 现在做
  → 否则排后面
```




## 五、面试中的产品能力展示（STAR框架）

```
Situation（背景）
  ├── 业务场景是什么
  └── 为什么这件事重要

Framework（框架）
 - 将现有动作按类型聚合成为方法论框架

Action（动作）
 - 描述核心是怎么干的
  
Result（结果）
 - 核心数据收益
```
