# 红包雨领金币玩法

## 【S - Situation 需求背景】

**业务验证**：
1. **定时领钱心智有效**：当前线上饭补、宝箱类玩法通过强预期的「定时领钱」心智，培养用户准点打开App的习惯，验证对DAU提升有效
2. **竞品验证**：抖音极速版、拼多多等通过随机性红包雨活动提升用户打开频次和时长，验证有效
3. **商业化延伸**：红包雨奖品除金币外，可联动电商等业务，串联广告场景与消费场景

**本期诉求**：通过红包雨与Push结合，进一步提升用户活跃频次

---

## 【T - Task 需求目标】

- **提升高价值用户DAU**
- **拉动广告曝光及电商转化**：通过红包雨串联广告场景和消费场景

---

## 【A - Action 需求详情】

### 一、红包雨任务入口

激励任务中心增加红包雨任务入口：

| 元素 | 设计 |
|:---|:---|
| **任务icon** | 红包雨icon |
| **主文案** | 金币红包雨，最多可得100雷 |
| **副文案** | 活动进行中：「金币红包雨，先到先得」；冷却中：「距离下一场开启: xx:xx:XX」 |
| **按钮** | 活动进行中：「领红包」；冷却中：「稍后领」 |

---

### 二、红包雨交互玩法

#### 1. 预热倒计时（3s）

- 全屏蒙层展示「红包雨即将降临」+ 3、2、1 倒计时

#### 2. 游戏过程

| 要素 | 设计 |
|:---|:---|
| **游戏时长** | 15s（支持分人群包配置，高价值用户15s） |
| **核心交互** | 屏幕上方不断掉落红包，用户点击红包 |
| **点击反馈** | 红包变为金币icon并消失，伴随点击音效和金币音效 |
| **掉落速度** | 随游戏时长逐渐变快 |

#### 3. 红包类型

- **红包大小**：大小不一致，大红包包含更多金币
- **高价值用户**：随机出现大红包，出现时机为游戏开启4s后，频率为单用户单日1个，n场次随机出现

#### 4. 炸弹元素

| 要素 | 设计 |
|:---|:---|
| **效果** | 点击炸弹，游戏时长 -1s |
| **标注** | 炸弹上标注扣除时间（-1s） |
| **出现时机** | 游戏开启3s后，50%概率出现 |
| **限制** | 单场最多出现1次 |

#### 5. 金币奖励计算

| 方案 | 逻辑 |
|:---|:---|
| **方案1** | 每个红包金币数量根据用户价值分层制定，用户实际获得 = 点击到的红包金币总和 |
| **方案2** | 用户获得 = (点击红包数 / 红包总数) × 该用户该场次可获得的reco |

#### 6. 挽留弹窗

- **触发**：游戏过程中点击左上角返回按钮
- **主文案**：确认要放弃一大包红包吗
- **副文案**：最高可得xx金币

---

### 三、结算弹窗

| 元素 | 设计 |
|:---|:---|
| **主文案** | 你在红包雨中获得{xx} kwai golds |
| **布局** | 左侧显示直接领取可获得的金币数量，右侧图示显示翻倍后可领取的金币数量 |
| **按钮A** | 看视频翻倍领取（看完激励视频金币可翻倍） |
| **按钮B** | 直接领取（直接发放当前数量的金币） |
| **领取反馈** | 金币发放成功后展示金币领取toast |

**副文案**：最高可得xx金币

**其他按钮**：继续抢红包、返回任务中心

---

### 四、异常情况

| 场景 | 处理 |
|:---|:---|
| **游戏过程中网络断开** | Toast提示：「网络已断开，请检查网络设置」 |

---

### 五、场次设置与Push

| 规则 | 说明 |
|:---|:---|
| **开启场次** | 每间隔10分钟开启一场，单用户单日最多10场 |
| **Push触发** | 红包雨开启时，使用强样式弹窗Push引导用户打开Kwai |
| **承接页** | 任务中心，打开后红包雨自动触发 |
| **弹窗屏蔽** | 红包雨触发时，需屏蔽其他任务中心弹窗 |
